## 1. 数据流与生命周期 ⬜

### 1.1 整体流程

```
Case 输入 → Provider 调用 → Evaluator 评估 → 结果聚合 → 持久化
     ↓           ↓              ↓             ↓          ↓
   校验失败    超时/失败      评估异常      部分失败    写入失败
```

### 1.2 待确定问题

- [ ] 每个阶段的**状态如何传递**？用 dataclass 还是 dict？
- [ ] 失败的 case 如何**标记和跳过**？状态字段设计？
- [ ] 断点恢复时如何**识别已完成的 case**？用 case_id + run_id 联合查询？
- [ ] Case 执行顺序：保持原顺序还是允许乱序？

### 1.3 备选方案

**方案 A：流式处理**
```python

    result = await process_case(case)
    await save_result(result)  # 即时保存
```

**方案 B：批量处理**

## 4. 错误处理架构 ⬜

### 4.1 错误分类

```python
class ErrorCategory(Enum):
    INPUT = "input"           # 输入数据问题
    PROVIDER = "provider"     # 模型调用问题
    EVALUATOR = "evaluator"   # 评估执行问题
    SYSTEM = "system"         # 系统级问题（IO、内存等）

class ErrorSeverity(Enum):
    WARNING = "warning"       # 可继续执行
    ERROR = "error"           # 当前 case 失败，继续下一个
    FATAL = "fatal"           # 必须中断整个 run

@dataclass
class EvalError:
    category: ErrorCategory
    severity: ErrorSeverity
    retryable: bool
    code: str                 # 错误码，如 "PROVIDER_TIMEOUT"
    message: str
    context: dict             # case_id, provider_name, etc.
    timestamp: datetime
    stack_trace: Optional[str] = None
```

### 4.2 错误码设计

| 错误码 | 分类 | 可重试 | 描述 |
|--------|------|--------|------|
| INPUT_INVALID_FORMAT | input | No | 输入格式不合法 |
| INPUT_MISSING_FIELD | input | No | 必填字段缺失 |
| PROVIDER_TIMEOUT | provider | Yes | 调用超时 |
| PROVIDER_RATE_LIMIT | provider | Yes | 被限流 |
| PROVIDER_AUTH_FAILED | provider | No | 认证失败 |
| PROVIDER_INVALID_RESPONSE | provider | Yes | 响应格式异常 |
| EVALUATOR_EXCEPTION | evaluator | No | 评估器内部错误 |
| EVALUATOR_INVALID_CONFIG | evaluator | No | 评估器配置错误 |
| SYSTEM_IO_ERROR | system | Yes | 文件读写失败 |
| SYSTEM_DB_ERROR | system | Yes | 数据库操作失败 |

- [ ] **错误恢复**：

  - 部分失败时，最终 run 状态是 SUCCEEDED 还是 FAILED？

## 5. SQLite Schema 设计 ⬜

### 5.1 核心表

```sql
-- 评测运行
CREATE TABLE runs (
    id TEXT PRIMARY KEY,
    dataset_path TEXT NOT NULL,
    provider_name TEXT NOT NULL,
    model_config JSON,
    concurrency INTEGER DEFAULT 1,
    timeout_ms INTEGER DEFAULT 30000,
    max_retries INTEGER DEFAULT 3,
    status TEXT DEFAULT 'pending',
    total_cases INTEGER,
    completed_cases INTEGER DEFAULT 0,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    started_at TIMESTAMP,
    finished_at TIMESTAMP,
    error_message TEXT
);

-- Case 结果
CREATE TABLE case_results (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    run_id TEXT NOT NULL,
    case_id TEXT NOT NULL,
    query TEXT,
    expected_answer TEXT,
    actual_output TEXT,           -- 可能很大，如何处理？
    passed BOOLEAN,
    score REAL,
    eval_type TEXT,
    eval_reason TEXT,
    latency_ms REAL,
    provider_status TEXT,
    error_code TEXT,
    error_message TEXT,
    retries INTEGER DEFAULT 0,
    tags JSON,                    -- 存 case 的 tags 用于聚合查询
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    FOREIGN KEY (run_id) REFERENCES runs(id),
    UNIQUE(run_id, case_id)       -- 断点恢复时防重复
);

-- 状态变更日志
CREATE TABLE state_logs (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    run_id TEXT NOT NULL,
    from_state TEXT,
    to_state TEXT,
    reason TEXT,
    timestamp TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    FOREIGN KEY (run_id) REFERENCES runs(id)
);

-- 聚合指标（计算后缓存）
CREATE TABLE run_metrics (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    run_id TEXT NOT NULL UNIQUE,
    total_cases INTEGER,
    passed_cases INTEGER,
    failed_cases INTEGER,
    error_cases INTEGER,
    pass_rate REAL,
    avg_latency_ms REAL,
    p50_latency_ms REAL,
    p95_latency_ms REAL,
    p99_latency_ms REAL,
    tag_metrics JSON,
    error_distribution JSON,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    FOREIGN KEY (run_id) REFERENCES runs(id)
);
```

### 5.2 待确定问题

- [ ] **大文本存储**：

- [ ] **JSON 字段 vs 拆列**：
  - `model_config`、`tag_metrics` 存 JSON 还是拆成独立列？
  - SQLite JSON 查询性能如何？

- [ ] **索引设计**：

- [ ] **数据保留**：

- [ ] **断点恢复**：

## 9. 对比功能 ⬜

### 9.1 对比输出结构

```python
@dataclass
class CompareResult:
    base_run_id: str
    candidate_run_id: str

    # 总体变化
    base_pass_rate: float
    candidate_pass_rate: float
    pass_rate_delta: float          # +0.05 表示提升 5%

    # 延迟变化
    base_avg_latency_ms: float
    candidate_avg_latency_ms: float
    latency_delta_ms: float
    latency_delta_pct: float        # 百分比变化

    # 按 tag 变化
    tag_changes: Dict[str, TagChange]

    # Case 级变化
    new_failures: List[str]         # base pass → candidate fail
    new_passes: List[str]           # base fail → candidate pass
    still_failing: List[str]        # 两次都 fail

    # 统计
    total_changed: int
    regression_count: int           # 变差的数量
    improvement_count: int          # 变好的数量
```
